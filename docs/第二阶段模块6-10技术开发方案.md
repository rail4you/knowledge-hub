# 第二阶段模块 6-10 技术开发方案

## 1. 文档目的

本文档用于指导合同附件中第二阶段 6-10 模块的设计与实施，目标是：

- 基于当前 `ABP Framework + .NET + Angular + PostgreSQL` 体系扩展，不拆新微服务；
- 尽量复用现有 `Resources`、`Search`、`Courses`、`Learning`、`Exams`、`Alliance`、`Identity`、`AI` 能力；
- 先形成可验收、可部署、可维护的标准业务能力，再逐步追加智能化和高并发优化；
- 明确本阶段的实现边界，避免把合同中的“理想目标”直接误写为首版必须一次性完成的全部代码范围。

## 2. 当前系统基础与可复用能力

当前仓库已经具备第二阶段开发所需的基础设施：

| 现有能力 | 当前位置 | 对第二阶段的复用方式 |
|---|---|---|
| 多租户/身份/角色权限 | `src/KnowledgeHub.Application.Contracts/Permissions/`、ABP Identity | 继续沿用 RBAC，新增模块权限组 |
| 联盟/院校协同 | `src/KnowledgeHub.Application/Alliance/` | 双高评估、微专业、就业统计按联盟/院校维度聚合 |
| 资源管理与附件上传 | `src/KnowledgeHub.Application/Resources/`、`ChunkUploadController` | 双高佐证材料、资讯封面、实训资料、简历附件统一复用 |
| 检索与索引 | `src/KnowledgeHub.Application/Search/` | 资讯检索、就业岗位检索、双高报表资料检索可复用 |
| 课程中心 | `src/KnowledgeHub.Application/Courses/`、`angular/src/app/learning/` | 微专业与实训均可建立在课程、章节、知识资源之上 |
| 学习进度 | `src/KnowledgeHub.Application/Learning/` | 微专业学习进度、实训学习留痕复用 |
| 习题与考试 | `src/KnowledgeHub.Application/Exams/` | 微专业考核、实训评分复用或扩展 |
| AI 职业规划 | `src/KnowledgeHub.Application/AI/CareerGuidanceAppService.cs` | 就业指导模块可直接嵌入 |
| 导出能力 | `ClosedXML`、已有 Docx 生成 | 双高报表、就业统计、证书导出可复用 |

结论：第二阶段不建议引入新的后端主框架，也不建议把核心链路拆到 Python 服务。Python 仅可作为离线数据处理或报表辅助工具，主业务仍以 .NET 为主。

## 3. 实施边界

### 3.1 本阶段纳入范围

- 6. 双高专业群评估支撑
- 7. 微专业
- 8. 就业
- 9. 资讯
- 10. 实训

### 3.2 本阶段暂不实现

- 实时视频直播
- 实时音视频面试
- 屏幕共享
- 大规模 RTC 并发调优
- 复杂 3D/仿真型实训引擎

### 3.3 替代方案

对合同中涉及实时音视频的部分，首版采用以下替代方式：

- 在线面试改为“面试预约 + 外部会议链接 + 面试记录 + 结果回填”
- 直播教学能力不纳入本次 6-10 模块开发范围
- 后续如需补齐，建议单独立项接入腾讯云 TRTC、声网 Agora 或自建 WebRTC 服务

## 4. 总体技术路线

### 4.1 分层设计

新增模块仍按当前仓库分层落地：

- `src/KnowledgeHub.Domain.Shared/<Module>/`：枚举、常量
- `src/KnowledgeHub.Domain/<Module>/`：聚合根、实体、仓储接口
- `src/KnowledgeHub.Application.Contracts/<Module>/`：DTO、`I*AppService`
- `src/KnowledgeHub.Application/<Module>/`：应用服务、领域编排
- `src/KnowledgeHub.EntityFrameworkCore/<Module>/`：EF Core 映射、仓储实现
- `src/KnowledgeHub.HttpApi/Controllers/`：仅在需要手写接口时增加
- `angular/src/app/<module>/`：独立页面、手写服务或生成代理调用

### 4.2 前端原则

- 优先使用 ABP 生成的 Angular proxy；
- 不手工改 `angular/src/app/proxy/**`；
- 契约变更后执行：

```bash
cd angular
abp generate-proxy -t ng
```

### 4.3 通用技术组件

建议统一复用以下能力：

- 文件上传：继续走 `ChunkUploadController` 与 `Resource` 附件体系
- 报表导出：`ClosedXML` 导出 Excel，必要时用 NPOI/DocX 导出 Word
- 定时/异步任务：ABP Background Jobs
- 审计日志：ABP 审计日志 + 模块业务日志表
- 多租户隔离：按 `TenantId` 处理院校侧数据，联盟管理员可跨租户聚合
- 搜索：首版用数据库筛选，增强版再接入 MeiliSearch

## 5. 模块开发优化级别定义

| 级别 | 含义 | 特征 |
|---|---|---|
| `L1 标准交付` | 满足功能验收 | CRUD、流程、权限、导出、日志、基础统计 |
| `L2 增强优化` | 提升运营效率 | 批处理、模板化报表、搜索优化、提醒、仪表盘 |
| `L3 智能增强` | 追加 AI/自动化 | 智能推荐、自动诊断、智能摘要、智能评分 |
| `L0 延后` | 本阶段不做 | RTC、直播、复杂仿真、超大并发专项 |

### 5.1 本轮建议目标级别

| 模块 | 首版目标 | 后续升级 |
|---|---|---|
| 双高专业群评估支撑 | `L2` | `L3` |
| 微专业 | `L2` | `L3` |
| 就业 | `L1` | `L2/L3` |
| 资讯 | `L2` | `L3` |
| 实训 | `L2` | `L3` |

说明：

- 就业模块首版先做岗位、简历、投递、预约面试、指导、统计，不做 RTC；
- 双高、微专业、资讯、实训均可在现有单体内达到较完整的 `L2`。

## 6. 模块 6：双高专业群评估支撑

### 6.1 业务目标

围绕“双高专业群”评估建立“指标管理、数据采集、佐证材料、分析汇总、报表导出”的闭环。

### 6.2 可复用能力

- 租户/联盟：按院校、联盟分层统计
- 资源模块：复用附件与佐证材料存储
- 搜索模块：后续用于检索评估资料
- 导出组件：复用 Excel/Word 导出

### 6.3 建议新增领域对象

建议模块命名：`DoubleHigh`

核心实体：

- `DoubleHighProject`：评估项目/申报批次
- `DoubleHighIndicatorCategory`：指标分类
- `DoubleHighIndicator`：指标项
- `DoubleHighIndicatorDataSource`：指标数据源配置
- `DoubleHighEvidence`：佐证材料
- `DoubleHighIndicatorValue`：指标采集值/快照
- `DoubleHighAssessmentTask`：采集/汇总任务
- `DoubleHighReport`：报表生成记录

### 6.4 关键实现点

- 指标树支持多级配置；
- 每个指标可绑定数据来源：
  - 手工填报
  - 系统自动汇总
  - 资源数量统计
  - 课程/微专业/实训/就业数据聚合
- 每次采集生成快照，不直接覆盖历史；
- 佐证材料可绑定多个资源或上传独立附件；
- 报表按模板导出 Excel/Word；
- 联盟管理员可查看汇总，院校管理员只看本校。

### 6.5 应用服务建议

- `IDoubleHighProjectAppService`
- `IDoubleHighIndicatorAppService`
- `IDoubleHighCollectionAppService`
- `IDoubleHighReportAppService`

核心接口：

- 项目 CRUD
- 指标树 CRUD
- 指标采集规则配置
- 手动采集/批量采集
- 佐证材料绑定与上传
- 项目汇总看板
- 报表导出与打印数据生成

### 6.6 前端页面建议

- `/assessment/double-high/projects`
- `/assessment/double-high/project/:id`
- `/assessment/double-high/project/:id/indicators`

- `/assessment/double-high/report-center`

### 6.7 权限建议

### 6.8 优化级别建议

- 后续：增加 AI 指标解读、缺口分析、自动摘要，升至 `L3`

## 7. 模块 7：微专业

### 7.1 业务目标

在现有课程中心基础上，增加“微专业”这一更高层级的培养单元，支持课程编排、学习、考核、证书发放与查询。

### 7.2 可复用能力

- `Course`、`Chapter`、`StudentCourse`
- `Exercise`、`Exam`
- `LearningProgress`
- `CareerGuidance` 可作为微专业职业路径辅导增强项

### 7.3 建议新增领域对象

建议模块命名：`MicroMajor`

核心实体：

- `MicroMajor`：微专业主体
- `MicroMajorCourse`：微专业与课程关联
- `MicroMajorEnrollment`：学生报名
- `MicroMajorCompletionRule`：结业规则

## 8. 模块 8：就业

### 8.1 业务目标

建立“企业发布岗位 - 学生维护简历 - 投递申请 - 面试预约 - 就业指导 - 去向统计”的完整链路。

### 8.2 本阶段边界

本阶段不做：

- 实时音视频面试
- 屏幕共享
- 会议流媒体录制

本阶段改为：

- 面试预约
- 外部会议链接
- 面试纪要
- 面试结果回填

### 8.3 可复用能力

- Identity 用户与角色
- 联盟/院校维度统计
- 资源上传用于简历附件、企业资料
- AI 职业规划用于就业指导

### 8.4 建议新增领域对象

建议模块命名：`Employment`

核心实体：

- `EmployerProfile`：企业档案
- `JobPosting`：岗位信息
- `JobApplication`：投递记录
- `StudentResume`：学生简历
- `InterviewSchedule`：面试预约
- `InterviewFeedback`：面试反馈
- `EmploymentGuidanceRecord`：指导记录
- `EmploymentOutcome`：就业去向

### 8.5 关键实现点

- 企业用户发布岗位，院校/联盟管理员可审核；
- 学生可维护多版本简历；
- 支持岗位检索、收藏、投递；
- 面试预约支持时间、地点、外部会议地址、备注；
- 面试结果可记录通过、待定、未通过；
- 嵌入 AI 职业指导入口，生成求职建议；
- 统计输出按院校、专业、年级、岗位类型、就业状态聚合。


对规范 AppService 统一走 ABP conventional controllers，变更契约后重新生成 Angular proxy。

### 11.4 审计与日志

每个模块至少记录：

- 创建/修改/删除
- 审核动作
- 导出动作
- 关键状态流转

### 11.5 报表导出

统一建设“导出中心”模式：

- 前端发起导出
- 后端生成导出记录
- 小文件同步导出，大文件可异步
- 记录导出人、导出时间、过滤条件

## 12. 建议开发顺序

### 12.1 推荐实施顺序

1. 双高专业群评估支撑
2. 微专业
3. 资讯
4. 实训
5. 就业

原因：

- 双高评估最依赖其他模块数据口径，宜先定义数据模型；
- 微专业与实训能最大化复用现有课程体系；
- 资讯实现独立，风险低，适合快速形成可见成果；
- 就业模块若排除 RTC，业务闭环并不复杂，但涉及企业端角色和统计口径，适合在后面收口。

### 12.2 推荐迭代划分

| 迭代 | 内容 |
|---|---|
| Iteration 1 | 权限、菜单、公共字典、导出基础、双高模块基础 |
| Iteration 2 | 微专业模块 |
| Iteration 3 | 资讯模块 |
| Iteration 4 | 实训模块 |
| Iteration 5 | 就业模块与统计收口 |
| Iteration 6 | 联调、修复、验收资料、部署脚本、培训文档 |

## 13. 风险与决策建议

### 13.1 合同指标与首版实现的差异

以下指标不应直接视为“代码层面一次性交付即可自动达标”，需要明确依赖基础设施和数据质量：

- 5 万并发在线
- 检索准确率 99%
- 数据采集准确率 100%
- 报表一小时内稳定生成

建议在验收文档中拆分为：

- 功能实现验收
- 数据口径验收
- 压测与部署能力验收

### 13.2 技术决策建议

- 不拆微服务，先保留单体；
- 不新增独立 Python 主业务服务；
- 所有业务模型先以可维护的结构化数据建模实现；
- 实时音视频单独列为后续专项；
- 第二阶段先做“标准化业务流 + 导出 + 审计 + 统计”，再补 AI。

## 15. 结论

在当前仓库基础上，模块 6-10 除实时音视频能力外，均可通过标准数据建模、ABP 应用服务、Angular 页面和常规导出统计方式落地。建议先按本文档进入逐模块开发，优先实现 `L1/L2` 验收范围，避免一开始把复杂直播、RTC、智能自动化全部打包进首版范围。
