# 第二阶段模块开发记录

## 记录目的

记录第二阶段模块的实际开发过程、关键决策、代码落点、构建结果与后续待办，便于后续继续开发和验收追踪。

## 2026-04-29 第一轮开发

### 本轮目标

- 按“先易后难”原则启动第二阶段开发；
- 优先实现依赖最少、业务边界最清晰的 `资讯模块`；
- 在不启动服务的前提下完成代码落地、编译检查与数据库迁移生成。

### 选择资讯模块的原因

- 与当前 `Resources`、`Courses`、`Learning` 等主链路耦合最低；
- 适合用标准 ABP CRUD + 审核 + 评论管控模式快速交付；
- 能尽快形成用户可见页面和后台管理页面；
- 可为后续“双高评估”和“就业”提供内容发布能力。

### 实施范围

本轮实现了模块 9 `资讯模块` 第一版，包含：

- 资讯分类管理
- 资讯文章管理
- 资讯审核状态流转
- 资讯前台列表
- 资讯详情查看
- 评论发布与后台管控
- 点赞能力
- 权限定义、本地化、菜单、路由
- EF Core 映射与数据库迁移

### 关键技术决策

#### 1. 先做手写 Angular 服务，不阻塞开发

由于本轮未主动启动 API 服务，未依赖 `abp generate-proxy -t ng` 完成前端接入，而是先使用手写 `RestService` 服务落地前端。

原因：

- 项目约束要求 agent 不自动启动服务；
- 先实现可运行页面比等待代理生成更高效；
- 后续若用户启动 API，可再切换为标准 proxy。

#### 2. 资讯模块首版采用“标准业务闭环”

首版未实现以下增强项：

- 资讯全文检索接入 MeiliSearch
- 分享日志
- AI 摘要/标签推荐
- 推荐位算法

保留为后续 `L3` 增强项。

#### 3. 评论采用“先展示、后管控”模式

首版评论默认直接显示，后台可执行：

- 显示
- 隐藏
- 拒绝
- 删除

这比首版就上评论预审核流程更简单，也满足“管理员后台管控”的合同要求。

## 2026-04-29 第二轮开发

### 本轮目标

- 继续按“先易后难”推进第二阶段开发；
- 优先实现可复用 `Courses`、`StudentCourse`、`Exams` 能力最多的 `微专业模块`；
- 在不启动服务的前提下完成后端、前端、权限、迁移与编译检查。

### 选择微专业模块的原因

- 与现有课程、学习进度、学生选课链路契合度最高；
- 首版无需引入复杂实时能力或外部系统；
- 能为后续实训、双高统计提供新的培养单元数据口径；
- 合同中的大部分功能可以先通过数据建模与常规业务流程实现。

本轮实现了模块 7 `微专业模块` 第一版，包含：

### 关键技术决策

#### 1. 微专业首版直接复用课程完成数据

首版不单独建设复杂学习引擎，而是直接复用：

- `Course`
- `StudentCourse`
- 课程完成状态

将微专业学习进度视为“已绑定课程完成度的聚合结果”。

原因：

- 能显著降低首版开发成本；
- 与现有学习链路保持一致；
- 后续要接入实训、综合考核时，也可以在当前聚合逻辑上继续扩展。

#### 2. 先不拆分多个 AppService

虽然技术方案中建议拆成：

- `IMicroMajorAppService`
- `IMicroMajorEnrollmentAppService`
- `IMicroMajorCertificateAppService`

但首版为减少接口分散，先集中在 `IMicroMajorAppService` 中实现微专业、报名、证书三个子域的核心能力，后续如接口规模继续扩大再拆分。

#### 3. 证书先做业务闭环，不做复杂模板化渲染

首版证书能力已实现：

- 证书编号
- 校验码
- 发放记录
- 我的证书查询

暂未实现：

- PDF/Word 正式版证书排版
- 对外公开查询页
- 证书模板可视化配置

以上能力保留为后续 `L3` 增强项。

## 2026-04-29 第三轮开发

### 本轮目标

- 在微专业完成后继续推进可复用度最高的 `实训模块`；
- 以“项目、任务、提交、指导、评分、导出”形成首版业务闭环；
- 继续遵守不自动启动服务的约束，只做代码、编译、迁移与文档更新。

### 选择实训模块的原因

- 与 `Courses`、`StudentCourse`、`Learning` 的复用关系最紧密；
- 可以直接承接微专业后续的综合实践场景；
- 合同验收点大多可以通过常规数据建模和流程控制完成；
- 不需要在首版引入 3D 仿真或实时互动能力。

### 实施范围

本轮实现了模块 10 `实训模块` 第一版，包含：

- 实训项目管理
- 任务拆解
- 实训资料配置
- 学生参与报名
- 学生按任务多次提交
- 教师指导记录
- 教师评分
- 学生过程时间线
- 成绩导出
- 权限定义、本地化、菜单、路由
- EF Core 映射与数据库迁移

### 关键技术决策

#### 1. 首版不拆多服务，先集中在一个 AppService 闭环

虽然技术方案建议拆成：

- `IPracticumProjectAppService`
- `IPracticumSubmissionAppService`
- `IPracticumAssessmentAppService`
- `IPracticumStatisticsAppService`

但首版为了加快交付，先集中在 `IPracticumAppService` 中实现项目、参与、提交、指导、评分、导出等能力。

原因：

- 接口依赖关系强，集中实现更利于快速验证；
- 首版接口数量虽然较多，但职责边界仍可控；
- 后续如统计和导出能力继续扩张，再拆分更合适。

#### 2. 实训进度按“任务提交完成度”聚合

首版未实现复杂过程引擎，而是用：

- 项目任务总数
- 学生已提交任务数

计算实训进度，并据此维护：

- `Enrolled`
- `InProgress`
- `Submitted`
- `Completed`

这样可以先满足验收中的“过程留痕”和“进度可追溯”。

#### 3. 导出先做标准 Excel

首版导出能力使用 `ClosedXML` 输出成绩表，包含：

- 实训项目
- 学生姓名
- 状态
- 进度
- 最终得分
- 最后提交时间
- 结项时间

暂未实现：

- Word/PDF 实训报告模板
- 项目成果汇编模板
- 统计大屏

以上保留为后续 `L3` 增强项。

### 关键技术决策

#### 1. 首版简化为“项目 + 指标 + 快照 + 佐证”模型

技术方案中原本建议增加：

- `DoubleHighIndicatorCategory`
- `DoubleHighIndicatorDataSource`
- `DoubleHighAssessmentTask`

但首版为了缩短交付路径，先将这些能力简化为：

- `DoubleHighProject`
- `DoubleHighIndicator`
- `DoubleHighIndicatorValue`
- `DoubleHighEvidence`
- `DoubleHighReport`

其中：

- 指标分类由 `CategoryName` 直接表达；
- 数据源配置由 `DataSourceType` 枚举表达；
- 采集任务调度先不抽独立实体，而由手动触发采集接口完成。

#### 2. 自动采集优先复用现有业务模块计数

首版自动采集支持以下数据来源：

- 资源数量
- 课程数量
- 微专业数量
- 实训项目数量
- 资讯数量
- 微专业报名量
- 实训参与量

原因：

- 这些数据在当前仓库中已经具备稳定实体和状态字段；
- 对双高评估场景来说，这类统计口径最先能产生实际价值；
- 可以快速验证“自动采集 + 快照留痕”的主流程。

#### 3. 手工填报和自动采集统一落快照表

无论是：

- 自动采集
- 手工填报

都统一写入 `DoubleHighIndicatorValue`，并带上：

- 采集来源
- 备注
- 采集时间

这样可以满足“每次采集生成快照，不直接覆盖历史”的设计目标。
